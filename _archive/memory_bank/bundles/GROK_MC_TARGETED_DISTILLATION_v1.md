### 1.3 Five Scoring Dimensions

#### A. Context Understanding (±2 points, weight: 25%)
- **Exemplary (2.0)**: Full problem space grasped, all dependencies mapped, user intent transparent
- **Adequate (1.0)**: Core context understood, some dependencies traced
- **Poor (0.0)**: Missing critical context, assumptions unvalidated
- **Example (Good)**: "Qdrant OOM because memory limit 1GB, Tier 2 services need 768MB pool, recovery includes WAL Tier-1 fallback"
- **Example (Bad)**: "Restart Qdrant. It should work." ← Missing diagnostics, no contingency

#### B. Decision Quality (±3 points, weight: 35%)
- **Exemplary (3.0)**: Optimal routing (right model), tool selection perfect, sequencing efficient, no rework needed
- **Adequate (1.5)**: Routing correct, some tool inefficiency, acceptable sequencing
- **Poor (0.0)**: Wrong model chosen, misaligned tools, chaotic ordering
- **Example (Good)**: "Use grep (15s) vs bash loop (4min). Haiku routes to GPT-4.1 for Phase 6 strategy (complexity>8). Sequence: Phase 0 user action → Phase 1 Haiku → Phase 5 handoff test → Phase 6 GPT-4.1"
- **Example (Bad)**: "Do everything with Haiku. Use bash for everything." ← No model differentiation

#### C. Execution Fidelity (±2 points, weight: 25%)
- **Exemplary (2.0)**: Spec-complete, edge cases handled, verification thorough, zero defects
- **Adequate (1.0)**: Core spec met, some edge cases handled, basic verification
- **Poor (0.0)**: Incomplete, missing edge cases, no verification
- **Example (Good)**: "Storage cleanup: Run 8 prune commands sequentially, verify each step, confirm disk <85% with `df -h /` output"
- **Example (Bad)**: "Clean up storage." ← No specifics, no verification

#### D. Documentation & Transparency (±1.5 points, weight: 10%)
- **Exemplary (1.5)**: Clear reasoning, explicit tradeoffs, failure recovery documented
- **Adequate (0.75)**: Reasoning present, some tradeoffs mentioned
- **Poor (0.0)**: No reasoning, no tradeoffs, opaque
- **Example (Good)**: "Fallback if WAL unrecoverable: Tier-2 recovery (3-4 hrs extra), data loss <0.1%, contingency: restore from backup"
- **Example (Bad)**: "If it fails, try again." ← No analysis

#### E. Ma'at Alignment (±1.5 points, weight: 5%)
- **Exemplary (1.5)**: Truth evident, Balance preserved, Advance demonstrated
- **Adequate (0.75)**: Some Ma'at resonance present
- **Poor (0.0)**: Conflicts with Ma'at principles

## Haiku→GPT-5-Mini→GPT-4.1 Handoff Mechanics & Checkpoint Strategy

### 6.1 Handoff Taxonomy

Three types of handoffs occur across the Omega execution:

#### Type A: Sequential Facet Handoff (Haiku → Mini → GPT-4.1)
**Use Case**: Multi-phase task requiring different reasoning depths
**Mechanism**: Checkpoint serialization + agent bus transmission
**Risk**: Memory loss, context deserialization errors
**Recovery**: Automatic retry (2x) + fallback to source facet

```
Phase 1-3: Haiku executes (speed needed)
    ↓ (checkpoint created)
Phase 3-4 Synthesis: GPT-5-Mini analyzes & plans (quality needed)
    ↓ (context + analysis transferred)
Phase 5+: GPT-4.1 owns execution (strategic depth needed)
    ↓
VALIDATION: Hash comparison (input ≟ output memory), zero-loss goal
```

#### Type B: Parallel Facet Engagement (Haiku + Gemini)
**Use Case**: Large-scale codebase work + independent verification
**Mechanism**: Haiku coordinates, Gemini investigates independently, results merged
**Risk**: Opinion divergence, redundant effort
**Recovery**: Decision override via Grok MC or user

```
Haiku: Execute refactoring (tactical execution)
   ║
   ╠══ Gemini: Verify patterns, find missed dependencies (async)
   ║
MERGE: Haiku incorporates Gemini findings
VALIDATION: Cross-check results, resolve conflicts, document rationale
```

#### Type C: Fallback Facet Activation (Primary → Secondary)
**Use Case**: Primary facet unavailable, stalled, or exceeding token limits
**Mechanism**: Automatic switch to secondary facet with full context transfer
**Risk**: Quality degradation, performance impact
**Recovery**: Return to primary facet when available

```
Haiku executing Phase 2 Qdrant recovery
   ↓ (stuck on diagnostic, token limit approaching)
Switch to GPT-4.1 (deeper reasoning on Qdrant internals)
   ↓ (GPT-4.1 completes diagnosis)
Handoff result back to Haiku for execution
VALIDATION: Verify solution quality, no regression
```

  Merge session history (new facet acknowledges prior decisions)

Implement Health Checks with Auto-Healing\nPodman v4.3+ introduced \"Healthcheck Actions,\" allowing the container to self-heal without external scripts.[1][2][3]\n\n*   **The `HealthOnFailure` Directive:** In your Quadlet file, set this to `kill` or `restart`.[3]\n*   **Systemd Integration:** When set to `kill`, Podman exits with a non-zero code when unhealthy.[6] If your `[Service]` section has `Restart=always`, systemd will immediately restart the container.\n\n**Example Quadlet (`~/.config/containers/systemd/db.container`):**\n```ini\n[Unit]\nDescription=Stateful Database Service[2][3][1]\n\n[Container]\nImage=docker.io/library/postgres:15\nVolume=/home/user/data:/var/lib/postgresql/data:Z[3][2][1]\n# Health check: check if postgres is accepting connections\nHealthCmd=pg_isready -U postgres\nHealthInterval=30s\nHealthRetries=3[1][2][3]\n# Action: Kill the container if it stays unhealthy, letting systemd restart it\nHealthOnFailure=kill[6][3][1]\n\n[Service]\n# Standard systemd restart logic\nRestart=always\nRestartSec=5s[2][3][1]\n\n[Install]\nWantedBy=default.target\n```[7][2][1][3]\n\n### 3. Ensure Persistence (Linger)\nBy default, rootless user services stop when the user logs out.[3] For stateful services, you **must** enable lingering so the processes start at boot and stay running.[3]\n```bash\nloginctl enable-linger $USER\n```[1][2][3][7]\n\n### 4. Best Practices for Stateful Services[3][8]\n*   **Volume Mounting:** Always use the `:Z` (private) or `:z` (shared) flags for SELinux relabeling on volumes to avoid permission denied errors in rootless mode.[2]\n*   **Auto-Updates:** Use `AutoUpdate=registry` in your Quadlet file.[2][9][1][3] This allows the `podman-auto-update.timer` to pull new images and restart the service only if the health check passes (if configured with `Notify=healthy`).[1][3]\n*   **Socket Activation:** For services like web servers, use systemd `.socket` units.[10][2] This allows the host to listen on a port

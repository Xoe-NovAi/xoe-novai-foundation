Multi-Stage Build Best Practices\nMulti-stage builds allow you to use multiple `FROM` instructions in a single `Containerfile` (or `Dockerfile`).[4][2][3][5] Each `FROM` starts a new stage, and you can selectively copy files from previous stages.[6][3][4]\n\n*   **Separate Build and Runtime:** Use a \"heavy\" image with all compilers and build tools (e.g., `golang:alpine`, `node:20`) for the build stage, and a \"minimal\" image (e.g., `alpine`, `scratch`, or `distroless`) for the final stage.\n*   **Named Stages:** Use the `AS` keyword to name your stages (e.g., `FROM node:20 AS builder`). This makes your `COPY --from=builder` instructions more readable.\n*   **Order for Caching:** Place instructions that change frequently (like `COPY . .`) as late as possible in each stage. Storage Efficiency & Layer Sharing\nPodman's architecture provides unique ways to optimize storage beyond standard multi-stage builds:\n\n*   **Shared Base Images:** When multiple applications share the same base image (e.g., a custom `mycompany/base-python:3.9`), Podman stores those base layers only once on the host.

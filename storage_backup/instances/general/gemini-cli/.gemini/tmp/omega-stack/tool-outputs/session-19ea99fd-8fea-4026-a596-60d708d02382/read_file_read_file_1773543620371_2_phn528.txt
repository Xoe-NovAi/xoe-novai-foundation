ROOT CAUSE ANALYSIS\n\n### The UID 100999 vs 1000 Mismatch: What and Why\n\n#### The Problem\n\nWhen Podman containers run without the `--userns=keep-id` flag, they operate in a \"user namespace\" where container-internal UIDs are mapped differently than the host:\n\n- **Container UID 0** (root inside container) → **Host UID 100999** (unprivileged on host)\n- **Container UID 1000** (regular user inside container) → **Host UID 101000** (unprivileged on host)\n\nWhen these containers create files (e.g., writing to `~/.gemini/memory_bank/`), they're owned by `100999` or `101000` on the host, but the actual host user (e.g., `uid=1000`) cannot read/write these files without explicit permissions.\n\n**Example**:\n```\nContainer writes to: /home/user/.gemini/memory_bank/data.json\nFile ownership on host: 100999:100999\nHost user: 1000:1000\nResult: Permission denied (13)\n```\n\n### Why Three Layers Are Required Together\n\nA single-layer approach fails because file permission problems are **multi-faceted**:\n\n| Layer | Scope | Problem It Solves |\n|-------|-------|-------------------|\n| **Layer 1: Ownership** | Existing files | Restores `1000:1000` ownership to files already created by containers |\n| **Layer 2: Default ACLs** | New files + directories | Ensures files created by containers are automatically readable by host user |\n| **Layer 3: Container Config** | Prevention at source | Prevents UID mismatch from happening in the first place |\n\n**Without Layer 1**: Existing corrupted files remain unreadable.  \n**Without Layer 2**: Future files created by containers will have wrong permissions again.  \n**Without Layer 3**: Even with Layers 1 and 2, containers not using `keep-id` will re-introduce the problem.\n\n### Why Layer 4 (Timer) Prevents Future Issues\n\nDespite best intentions, permissions can regress:\n\n- **ACL masks are recalculated** when files are chmod'd (even innocently by deployment scripts)\n- **Containers may restart** with different namespace configs\n- **New mount points** might be added without proper ACL defaults\n- **Team members** might forget to apply all three layers when adding new paths\n\n**Layer 4 provides continuous enforcement**: A systemd timer runs daily (or on demand) to:\n- Detect ownership mismatches\n- Restore correct ACLs automatically\n- Log all corrections for audit trail\n- Alert when critical paths are affected\n\nThis transforms the problem from \"reactive firefighting\" to \"proactive health monitoring.\"\n\n---\n\n## 2.

This is particularly relevant for the Xoe-NovAi Foundation stack running on Ryzen hardware with integrated Graphics (iGPU), where memory contention between the CPU and GPU is a primary performance bottleneck.\n\n---\n\n## 🔬 Multi-Device zRAM Architecture\n\n### 1. The Tiered Swap Concept\nLinux allows multiple swap devices with different priorities (`swapon -p`). By creating multiple zRAM devices, we can optimize for both latency and capacity.\n\n| Device | Algorithm | Priority | Target Usage |\n|--------|-----------|----------|--------------|\n| `/dev/zram0` | `lz4` | 100 | **High Frequency**: Small, fast pages, system buffers, interactive UI. |\n| `/dev/zram1` | `zstd` | 50 | **Bulk Capacity**: Model weights, inactive background processes, large context chunks. |\n\n### 2. Ryzen & iGPU Specific Optimization\nThe Ryzen iGPU (Radeon) shares system RAM via the UMA (Uniform Memory Access) buffer.\n-   **The Contention**: When the iGPU claims RAM, the CPU has less physical headroom, forcing earlier swapping.\n-   **The Solution**: Use a high-priority `lz4` zRAM device to minimize the \"stutter\" felt when the iGPU and CPU compete for bandwidth. The `lz4` algorithm has lower CPU overhead, leaving more cycles for iGPU drawing/inference.\n\n---\n\n## 🏗️ Experimental Project: \"Project Multi-ZRAM\"\n\n### Goal\nImplement a dual-device zRAM setup and measure the impact on RAG API latency under heavy load.\n\n### Proposed Configuration\n-   **ZRAM_FAST**: 1GiB, `lz4`, Priority 100.\n-   **ZRAM_BULK**: 4GiB (or 50% of remaining), `zstd`, Priority 50.\n\n### Implementation Script (PoC)\n```bash\n# Fast Tier\nzramctl --find --size 1G --algorithm lz4\nmkswap /dev/zram0\nswapon -p 100 /dev/zram0\n\n# Bulk Tier\nzramctl --find --size 4G --algorithm zstd\nmkswap /dev/zram1\nswapon -p 50 /dev/zram1\n```\n\n---\n\n## ⚖️ Best Practices & Pitfalls\n\n### Pitfalls\n-   **CPU Overhead**: Over-compressing with multiple `zstd` streams can starve the CPU during inference.\n-   **Memory Overhead**: Each zRAM device has a small management overhead in the kernel.\n-   **Context Drift**: If the fast tier is too small, the system constantly swaps between tiers, increasing latency.\n\n### Best Practices\n-   **Stream Alignment**: Set `max_comp_streams` to match the number of physical cores (not threads) to avoid cache thrashing.\n-   **Swappiness Tuning**: Use `vm.swappiness = 180` to ensure the system utilizes the tiers before touching physical disk (if any).\n-   **Page Cluster**: Use `vm.page-cluster = 0` to reduce I/O latency for single-page reads.\n\n---\n\n## 🚀 Future Roadmap: Dynamic Steering\nDevelop a \"Sovereign Governor\" that monitors iGPU usage and dynamically adjusts zRAM priorities or sizes to prevent system locks.\n\n---\n\n**Next Steps**: Implement the robust initialization script and verify the single-device baseline before moving to Multi-ZRAM.\n"
